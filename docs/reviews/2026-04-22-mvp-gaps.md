## Executive Summary

The architecture and implementation plans are comprehensive and well-structured. However, several gaps and risks could impact Phase 1 delivery or require early mitigation. This review identifies the **top 5 concerns** that should be addressed before or during MVP development.

---

## 1. Signal Pipeline Scope Mismatch (HIGH RISK)

### Gap
The architecture diagrams show 4 signal sources (Benzinga, Polygon, SEC EDGAR, Alpaca), but the implementation plan states: *"start with Alpaca market data only for MVP"*. 

### Risk
- **Agents will lack news/sentiment signals** — the core value proposition relies on agents reacting to market-moving news, not just price data
- Alpaca market data alone cannot provide sentiment, novelty, or materiality scores that the enrichment pipeline is designed to produce
- The Haiku enrichment step (sentiment/novelty/materiality) has no meaningful input with market data alone

### Recommendation
- **Option A:** Add Benzinga news feed to Phase 1 (implementation plan Task 2.6) — this is the minimum viable signal source for meaningful agent decisions
- **Option B:** Document that Phase 1 agents will operate on price data only, with signal-driven decisions deferred to Phase 1.5
- Define fallback behavior when signal sources are unavailable

## 2. Error Recovery & Partial Failure Handling (HIGH RISK)

### Gap
The implementation plan mentions Inngest durability but lacks detailed error recovery flows for partial failures in the decision-to-execution pipeline.

### Specific Scenarios Not Addressed
1. **Decision approved → Order submission fails** — Does the decision status roll back to `proposed`? Is the user notified?
2. **Order submitted → Fill never received** — Reconciler polls, but what's the timeout? What state does the decision land in?
3. **Constraint check passes → LLM retry exhausted** — What happens to the decision record?
4. **Push notification fails** — Is there a fallback (in-app notification, Supabase Realtime)?

### Recommendation
- Add explicit state transition diagrams for error paths in each Inngest workflow
- Define max retry counts and timeout thresholds for each step
- Document compensation actions (e.g., cancel order if decision rejected during execution window)
- Add dead-letter handling for workflows that exhaust retries

## 3. Testing Strategy Underspecified (MEDIUM-HIGH RISK)

### Gap
The implementation plan includes unit tests for schemas and constraints but lacks coverage for:
- **Integration tests** for the full decision flow (trigger → context → LLM → constraints → execution)
- **Eval harness** shown in architecture diagrams (§6.5, §11.5) but not included in implementation tasks
- **Load/stress testing** for concurrent agent decisions
- **Test coverage thresholds** not defined

### Risk
- Regressions may slip into production without integration coverage
- LLM behavior changes (prompt drift, model updates) won't be caught without the eval harness
- CI pipeline may pass while critical flows are broken

### Recommendation
- Add Task 2.7: "Create eval harness with golden decision scenarios" — this is shown in the diagrams but missing from the plan
- Define coverage requirements: ≥80% for `packages/core`, `packages/agent/constraints`
- Add one integration test per Inngest workflow in Phase 3
- Include smoke test for decision flow in CI (can use mock LLM responses)

## 4. Observability Gaps (MEDIUM RISK)

### Gap
MVP explicitly defers Langfuse and Grafana dashboards, leaving limited visibility into:
- **LLM call patterns** — no trace viewer beyond raw Postgres queries
- **Cost burn rate** — `admin.getUsage` endpoint exists but no alerting
- **Constraint failure patterns** — no aggregated view of why decisions are rejected
- **Drift detection** — `drift.ts` exists but no alerting mechanism defined

### Risk
- Cost overruns may not be detected until end-of-month billing
- LLM misbehavior (hallucination, refusals) invisible without trace analysis
- Debugging production issues requires manual SQL queries

### Recommendation
- Add minimal dashboard page to mobile app (Settings → Debug) showing:
  - Today's LLM cost (sum from traces table)
  - Last 24h constraint failures by layer
  - Agent decision counts
- Configure Sentry alerts for error rate thresholds (e.g., >5% decision failures)
- Add daily cost summary to #e8e-folio-dev via Slack webhook (can reuse existing infrastructure)

## 5. Authentication & Key Management (MEDIUM RISK)

### Gap
- Broker API keys stored in `expo-secure-store` on device — rotation strategy deferred to Phase 2
- No documented process for key revocation if device is compromised
- Rate limiting only mentioned for server-side (tRPC middleware) — mobile client can still spam requests before server throttling kicks in

### Risk
- Compromised device could drain paper trading account or leak API keys
- No circuit breaker if mobile client is sending malformed requests at high rate
- Key rotation in production will require app update if not designed now

### Recommendation
- Store only short-lived session tokens on device; keep broker keys server-side only
- Add client-side request throttling (debounce rapid approvals, max 1 request/sec)
- Document "device compromise" runbook: how to revoke tokens, disable agent, rotate keys
- Consider device attestation for high-risk actions (kill switch toggle)

## Additional Concerns (Lower Priority)

### 6. Generative UI Complexity
The 30-60s cache TTL + streaming layout composition adds significant complexity. For MVP, a static layout with data-driven content may reduce risk and development time.

### 7. Cross-Agent Conflict Detection
L2 constraint mentions "cross-agent check" but implementation details are sparse. If one user has multiple agents targeting the same ticker, how are conflicts resolved?

### 8. Approval Timeout Handling
`approval_expires_at` field exists, but the workflow that transitions expired decisions to `status='expired'` is not explicitly defined. This needs an Inngest scheduled job.

### 9. Offline Write Handling
MMKV caching handles offline reads, but what happens if a user approves a decision while offline? The mutation will fail — is this surfaced clearly in the UI?
